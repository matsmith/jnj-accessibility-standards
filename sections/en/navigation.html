Navigation

Keyboard accessible

Keyboard users navigate web pages sequentially by moving from one focusable element (such as a link or form control/input) to the next. It is essential, therefore, that all functionality is implemented in a way that allows keyboard users to focus and activate any "trigger" elements on the page, otherwise they may be prevented from navigating between pages or completing specific tasks.

Close button does not receive focus

The close "X" used to back out of the Balance coaching is not reachable via keyboard. Screen reader and keyboard only users will not be able to interact with this control. Keyboard only users, in particular, will find this frustrating since they will be able to see the control on the screen that they will not be allowed to interact with.

Dialog close button

The "X" control used to dismiss dialogs is not reachable via keyboard. To compound the issue, these dialogs are not able to be dismissed via the ESC key which is a common design pattern for this type of component.

Recommendations

Ensure that all actionable elements are reachable and can be operated using the keyboard alone, and that elements that are not actionable are not included in the keyboard tab order.

Ideally, button elements should be created using native HTML elements that expose a role of button (input with a type of button, submit or image, or a button element), as these elements also work with the correct keyboard behavior. The browser then handles all the keyboard support for native elements.

Related Standards

WCAG 2.0 – 2.1.1 Keyboard (Level A)

All functionality of the content is operable through a keyboard interface without requiring specific timings for individual keystrokes, except where the underlying function requires input that depends on the path of the user's movement and not just the endpoints.

WCAG 2.0 – 4.1.2 Name, Role, Value (Level A)

For all user interface components (including but not limited to: form elements, links and components generated by scripts), the name and role can be programmatically determined; states, properties, and values that can be set by the user can be programmatically set; and notification of changes to these items is available to user agents, including assistive technologies.

Bypass blocks

Navigation links are the lists or "menus" of links to all the sections of a site that are often repeated on every page. As navigation links are typically placed at the beginning (top) of pages, screen reader users and sighted keyboard-only users are often required to read through all the navigation links before reaching the main area of the page.

Most tested pages did not have skip links, while others had them but they were not shown on focus. This adds an unexplained focus stop to keyboard only users and deprives them of functionality that they would otherwise find helpful.

Recommnedations

The WAI ARIA specification defines a set of specialized landmark roles. These roles provide a method to programmatically identify commonly found regions of web content in a consistent way, and allow assistive technologies to provide users with features they can use to identify and navigate to sections of the page.

The following landmark roles can be used:

  • banner
  • complementary
  • contentinfo
  • main
  • navigation
  • search
  • application

To add landmark regions to an HTML document, identify the logical structure of the document. All content should be contained within a region so that screen reader users can be sure to get to all content, and not miss orphaned content. Add the landmark to the element containing the area of the content as a role attribute with the value corresponding to the list of roles above.

For example, identify the main (unique) content on the page by adding role="main" to a containing <div> element.

Related Standards

WCAG 2.0 – 2.4.1 Bypass Blocks (Level A)

A mechanism is available to bypass blocks of content that are repeated on multiple Web pages.

Page and iFrame titles

Incorrect page titles within Single Page Application

While some pages do correctly use the <title> element, many of the tested pages lacked unique or meaningful titles, relying instead on generic section titles such as "Digital Health Coaching."

Titles missing for iframes

Properly titled iFrames help users quickly navigate and understand the content of an iFrame without having to interact with the content. When these frames are not properly titled, their purpose or meaning is not properly conveyed to the user.

Recommendations

Using the <title> element, make sure that each page has a unique page title that adequately summarizes the page content. For helpful wayfinding, a general rule of thumb is to use the following format: Page title – Section title – Application title. This format describes the current page as well as the current content, providing a "you are here" function.

Related Standards

WCAG 2.0 – 2.4.2 Page Titled (Level A)

Web pages have titles that describe topic or purpose.

Focus and document flow

Keyboard navigation involves using certain keys to move keyboard focus around the elements on the screen, and other keys to select the element that currently has keyboard focus. When keyboard commands trigger unexpected actions, the resulting change of context is disorienting.

There are few notable instances in the tested views where focus is not properly managed, leading to either a loss in focus or the user being able to navigate away from an area of the page when they should not be able to.

When a dialog is opened, focus is not moved to the window, instead the user is able to navigate around the page behind it.

Recommendations

Avoid attaching unexpected behaviors to controls. Unless change is expected and a direct result of user input, focus should remain on the trigger element and move forward as expected.

If a user opens a dialog, focus should be returned to that control when the dialog is dismissed. When loading additional information, such as additional articles generated by a "Read More" button, focus should move to the first new article.

When focus is moved to a menu that the user has opened, focus should remain inside that menu until the user makes a selection or closes it with the ESC key.

The non-keyboard specific portions of the WAI-ARIA design patterns for menus and dialogs must be adhered to. For information on the expected behavior refer to:

  • Dialog modal
  • Menu

Visible focus

Sighted keyboard-only users navigate websites by moving focus sequentially through the page (using TAB/SHIFT+TAB to move the focus from one focusable element – such as a link or form control/input – to the next). While the text is underlined on hover, on many of the pages that were tested, indication of focus for keyboard users (with the exception of global nav and footer) falls back to the browser default. This makes it difficult for this user group to determine where their cursor currently is within the content.

It is imperative that interactive elements such as links, button, tabs and form select combo box controls clearly indicate when they have focus (i.e. focus indication should have high visual contrast). Otherwise sighted people who use a keyboard to interact with the website will not be able to determine their current position within the page.

Throughout all tested pages there were significant issues with visible focus. In some cases, the browser default was used which is often insufficient. And in others there was no indication at all.

Recommendations

Provide a clearly visible indication of focus for all interactive elements. Developers that use CSS for presentation will typically use the hover pseudo class for indicating that the mouse is over an element, but overlook similar functionality for keyboard only users. For anchor and input elements, this is simply a case of including a focus pseudo class to provide the same information mouse users get when they hover over an element.

Any styles that applied, or changed on :hover must be also be observed on :focus.

For example where a CSS rule is set to indicate mouse hover on a link, add a: active (due to a bug in earlier versions of IE (before version 8), the active pseudo class must also be used) and a:focus, which will result in the same style being applied on focus.

The following recommendations should be applied together:

  • Do not use CSS outline: none to hide the default focus indicator.
  • Provide indications of focus in the same manner and style that indications of mouse hover are provided.
  • Provide an indication of focus for input type="image" and links containing only an image.

Related Standards

WCAG 2.0 - 2.4.7 Focus Visible (Level AA)

All user interface elements that receive focus must have a visible focus indicator.

Focus and reading order

Users that navigate with a keyboard (both sighted users and users with assistive technology such as screen readers) will generally use Tab/Shift+Tab to move sequentially from one focusable element in the page to the next. By default, the order in which elements receive focus is determined by the order in which these elements are present in the document's markup. If this source order is not logical, users will have difficulty understanding the purpose/structure of the document and may miss important information and functionality.

While most of My HealthMedia follows a logical and intuitive order, there were noticeable instances where reading order deviates from this standard.

Recommendations

Make sure that the order in which elements receive keyboard focus, or are discovered by the user, follow a meaningful and logical sequence. Pay attention to code order; the tab order must make sense visually, but also match the HTML element order. Sequencing elements in the code so they follow a logical order helps to provide a logical and meaningful focus order.
